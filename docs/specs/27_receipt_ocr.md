# レシートOCR入力補助 仕様書

## 概要

収支登録フォームで、レシート画像から**合計金額**と**日付**を読み取り、金額欄・日付欄に
自動入力（プリフィル）する機能。手入力の手間を減らすことが目的。

要件は**完全無料・LLM 不使用**。OSS の OCR エンジン **Tesseract.js**（WASM 版 Tesseract、
日本語 `jpn`）を**ブラウザ内（クライアントサイド）**で実行し、認識テキストを正規表現＋
優先度ルールで解析して金額・日付を抽出する。サーバー計算・ストレージ・外部 API を使わないため
運用コストはゼロ。

OCR 精度は保証しない。抽出結果はあくまで入力補助であり、**ユーザーが内容を確認・修正してから
登録する**前提とする。レシート画像は保存せず、認識後に破棄する。

## 対象ユーザー・前提条件

- ログイン済みかつアクティブな household に所属しているユーザー。
- 対象は収支の新規登録・編集フォーム（同一の `TransactionForm`）。
- マイグレーション不要（DB スキーマ・RLS は変更なし）。画像は保存しない。
- `jpn.traineddata` は初回のみ CDN から取得され、以降は IndexedDB にキャッシュされる
  （初回は数秒のダウンロードが発生）。

## 抽出仕様

解析ロジックは純粋関数 `lib/receipt/parse-receipt.ts` に集約する（ブラウザ非依存・単体テスト可能）。

```ts
parseReceipt(rawText: string, today?: Date): { amount: number | null; date: string | null }
```

- **正規化**: 全角の数字・記号（`０-９`、`，．￥－ー／`、全角空白）を半角へ変換する。
- **金額**:
  - 合計キーワード行を優先抽出する。優先度 1: `合計` `お会計` `ご請求` `請求金額` / 優先度 2:
    `お買上` `総計` `total`。`小計` `中計` を含む行は除外する。
  - キーワード判定は OCR のスペース混入に備え空白除去後に行う。該当行の**右端の数値**を金額とする。
  - キーワードが無ければ、通貨らしいトークン（`¥`前置・`円`後置・カンマ区切り）の**最大値**を
    フォールバックにする（電話番号などの非通貨数値を無視するため）。
  - 範囲ガード: 1〜9,999,999 円。範囲外・未検出は `null`。
- **日付**:
  - 年付き `YYYY[区切り]MM[区切り]DD`（区切りは `/ - . 年 月 日`）を最優先。年は 2000〜翌年で検証。
  - 年が無ければ `MM/DD`（区切り `/ - 月 日`）を `today` の年で補完。コロン区切りの時刻は誤認しない。
  - 月 1〜12・日 1〜31 を検証し、`YYYY-MM-DD` で返す。未検出は `null`。
  - 和暦（令和など）は対象外。
- OCR 実行は `lib/receipt/ocr.ts` の `recognizeReceipt(file, onProgress)` に集約。`tesseract.js` を
  関数内で**動的 import** し、巨大画像は canvas で長辺 2000px まで縮小してから認識する。

## 画面・UI

- 収支登録フォーム上部に「レシートを読み取る」ボタン（`ReceiptScanButton`）を配置。
  説明文「レシート画像から金額・日付を自動入力します（内容をご確認ください）」を添える。
- ボタン押下で画像選択（`accept="image/*"` / `capture="environment"` でモバイルはカメラ起動可）。
  認識中はボタンに進捗（`読み取り中… NN%`）を表示し、二重実行を防ぐ。
- 重い Tesseract.js は `next/dynamic`（`ssr:false`）の `receipt-scan-button.client.tsx` 経由で
  遅延ロードし、ボタン描画までライブラリを取得しない（Recharts と同方針）。
- 抽出後の挙動（`TransactionForm`）:
  - 金額が取れれば金額欄（uncontrolled）に設定し、金額プレビューを更新する。
  - 日付が取れれば日付欄に設定する。取れた項目のみ反映し、他は既存値を維持する。
  - `sonner` トーストで結果を通知（成功時は抽出値の要約、未抽出時は手入力を促す、失敗時はエラー）。

## バリデーション

- 抽出結果は金額欄・日付欄に**文字列として投入するだけ**で、検証は既存フローを正とする。
  - 金額は既存 zod スキーマ（`lib/validations/transaction.ts`、四則演算 preprocess 含む）で検証。
  - 日付は `type="date"` の `required` で検証。
- OCR で読み取れない／誤読がありうる前提で、登録前にユーザーが確認・修正する。

## テスト

- `lib/receipt/parse-receipt.test.ts`（Vitest）: 金額（合計/小計混在・カンマ・全角・¥/円・フォールバック・
  範囲ガード）と日付（YYYY/MM/DD・年月日・ハイフン・年補完・不正日・時刻誤認）を網羅。
- `components/features/transactions/receipt-scan-button.test.tsx`: `ocr.ts` を mock して canned テキストを返し、
  `onResult` が正しい `{amount, date}` で呼ばれること、未抽出・例外時の挙動を検証（Tesseract 自体は動かさない）。
- 手動 E2E: `/transactions/new` で実レシートを読み取り、金額・日付がプリフィルされ、修正のうえ登録できること。

## 未解決の課題

- Tesseract の日本語レシート（感熱紙フォント）の認識精度は限定的。確認・修正前提を UI で明示している。
- 和暦の日付・税抜/税込の判別・店名（メモ）抽出は未対応（将来拡張）。
- `jpn.traineddata` の初回ダウンロード容量とオフライン時の挙動。将来セルフホストを検討。
